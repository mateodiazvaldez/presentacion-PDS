03 — Patrones GoF

Abstract Factory

Creacional

Problema: Argentina y Brasil tienen impuestos, vehículos y reglas de negocio distintos.

La lógica de negocio no puede estar llena de if (país == "ARG"). Abstract Factory encapsula toda la variación regional.

IFactoryRegional
├── FactoryArgentina → VehiculoNacional, ImpuestoIIBB
└── FactoryBrasil → VehiculoEstadual, ImpuestoICMS
Beneficio clave

Agregar una nueva región = implementar una nueva Factory.

El controlador no cambia.

16 / 29
03 — Patrones GoF

Builder

Creacional

Problema: Envio tiene 10+ atributos opcionales. Un constructor con 10 parámetros es ilegible e inutilizable.

Builder separa la construcción del objeto de su representación. Permite crear envíos paso a paso con solo los campos necesarios.

EnvioBuilder — Fluent API
new EnvioBuilder()
  .conOrigen("Buenos Aires")
  .conDestino("Córdoba")
  .conPeso(5.2)
  .conUrgencia(true)
  .conSeguro(500.0)
  .build()

Prototype implementado en Envio.clonar() para duplicar envíos recurrentes.

03 — Patrones GoF

Template Method

Comportamiento

Problema: El proceso de un envío siempre sigue el mismo esqueleto. Solo algunos pasos cambian.

ProcesadorEnvio.procesar():
  1. validar()      ← abstract
  2. calcularCosto() ← abstract
  3. procesarPago() ← concreto
  4. notificar()   ← abstract

Cada tipo de envío (nacional, internacional, express) hereda y redefine solo lo que cambia.

Decorator — Extensión sin herencia

Agrega comportamiento a un envío sin modificar su clase: seguimiento GPS premium, notificación SMS, seguro adicional.

EnvioDecorator
├── SeguimientoGPS
└── NotificacionPremium

04 — Capa de Persistencia

Data Mapper + Repository

Separación de preocupaciones
Data Mapper
Dominio puro

Las clases Envio, Cliente no tienen SQL. EnvioMapper y ClienteMapper trasladan datos entre objetos y tablas.

Cambiar la base de datos no toca ni una línea del modelo de negocio.
Repository
Interfaz uniforme

La lógica de negocio llama a RepositorioEnvios.buscar(id). No sabe si es SQL o memoria.

RepositorioEnvios (interfaz)
├── RepositorioEnviosSQL (producción)
└── RepositorioEnviosMemoria (tests)
23 / 29
04 — Capa de Persistencia

Unit of Work

Consistencia transaccional

Problema: Crear un envío requiere 3 escrituras. Si falla la tercera, la base queda inconsistente.

unitOfWork.registrarNuevo(envio)
unitOfWork.registrarNuevo(pago)
unitOfWork.registrarModificado(centro)
unitOfWork.commit()  ← todo o nada
Dirty tracking
NEW Entidades nuevas
DIRTY Entidades modificadas
DEL Entidades eliminadas

Un solo commit() sincroniza todo en la misma transacción.

24 / 29
04 — Capa de Persistencia

Lazy Load

Rendimiento

Problema: Cargar un envío instancia toda la jerarquía del cliente. Demasiado si solo necesitamos el nombre.

ClienteLazyProxy {
  cargado = false
  getNombre() {
    if (!cargado) cargarDesdeDB()
    return nombre
  }
}

El proxy es transparente. El cliente de dominio no sabe que existe.

Los 4 patrones como sistema
→ Data Mapper separa dominio y SQL
→ Repository abstrae el almacenamiento
→ Unit of Work garantiza atomicidad
→ Lazy Load optimiza la memoria
